## Global principles

### 1. Tokens-first

Never hardcode colors, spacing, font-sizes, radius, or shadows. If you find yourself typing a hex code or a `px` value, stop and find the token.

```css
/* ✗ bad */
.card { padding: 16px; background: #2563eb; }

/* ✓ good */
.card { padding: var(--kern-spacing-4); background: var(--kern-color-action-primary); }
```

### 2. Semantic over primitive

Reach for `color.action.primary` before `color.blue.600`. Semantic tokens communicate intent and survive palette changes in v2.

| Avoid | Prefer |
|-------|--------|
| `color.blue.600` | `color.action.primary` |
| `color.gray.900` | `color.text.default` |
| `color.gray.50` | `color.surface.subtle` |
| `color.red.600` | `color.feedback.error` |

### 3. One source of truth

`tokens/` and `specs/` are the only files you ever hand-edit. Everything in `build/`, `skill/`, and `llm-context/` is generated by `npm run build`. Never edit generated files.

### 4. Composition over invention

Combine existing components and tokens before designing new variants. A new variant should solve a generalizable problem, not a one-off layout need.

### 5. Accessibility is not optional

Every interactive component must be:
- Keyboard-operable (Tab, Enter, Space)
- Focus-visible (2px solid focus ring with offset)
- Screen-reader-friendly (correct roles, labels, and ARIA attributes)
- **Color alone never communicates state** — always pair color changes with text or icons

### 6. Predictable spacing rhythm

Stick to the spacing scale. Custom values create visual noise and make spacing inconsistent across breakpoints. When in doubt, use the next step up on the scale rather than a custom value.

### 7. Hierarchy through scale, not weight

Use font-size and spacing to establish hierarchy before reaching for bold or color emphasis. The type scale provides enough range — resist adding custom sizes.

## Composition patterns

### Card + Button
Put primary actions inside `CardFooter`, not floating in `CardBody`. This creates a consistent action zone at the bottom of every card.

```jsx
// ✓ good
<Card>
  <CardBody><p>Content</p></CardBody>
  <CardFooter><Button variant="primary">Primary action</Button></CardFooter>
</Card>

// ✗ bad — action buried in body
<Card>
  <CardBody>
    <p>Content</p>
    <Button>Action</Button>
  </CardBody>
</Card>
```

### Input + Form
Always pair Input with a visible `<label>` via the `label` prop. Show `errorMessage` after submit or on blur — not before the user has interacted with the field.

```jsx
// ✓ good — error shown after interaction
const [touched, setTouched] = useState(false);
<Input
  id="email"
  label="Email address"
  onBlur={() => setTouched(true)}
  errorMessage={touched && !isValid ? 'Enter a valid email' : undefined}
/>
```

### Button groups
Pair one primary Button with one secondary or ghost for cancel. Never place two primary buttons side-by-side — it removes hierarchy and forces the user to decide between two equally-weighted actions.

```jsx
// ✓ good — one primary, one ghost
<Button variant="primary">Save</Button>
<Button variant="ghost">Cancel</Button>

// ✗ bad — two primaries
<Button variant="primary">Save</Button>
<Button variant="primary">Save & continue</Button>
```

## Per-component rules

### Button

**Do**
- Use one primary Button per section — it signals the single most important action
- Label with a verb that describes the outcome: *Save changes*, *Create project*
- Use `loading` for async operations lasting more than 300ms
- Pair primary with secondary or ghost for cancel/back

**Don't**
- Don't use Button for navigation — use `<a>` instead
- Don't stack multiple primary Buttons in the same view
- Don't use `disabled` to hide unavailable actions — explain why it's unavailable
- Don't use the ghost variant for the most critical action

---

### Input

**Do**
- Always provide a visible label via the `label` prop
- Use `helperText` to set expectations before the user interacts
- Show `errorMessage` after validation — not before interaction
- Use `type="email"`, `type="password"` for correct mobile keyboards and auto-fill

**Don't**
- Don't use `placeholder` as a label — it disappears and fails contrast requirements
- Don't show errors before the field has been touched
- Don't rely on color alone to communicate error state
- Don't use for multi-line text — use `<textarea>` instead

---

### Card

**Do**
- Group a coherent set of related information — a product, a person, a task
- Use `CardHeader` / `CardBody` / `CardFooter` for consistent hierarchy
- Use `as="article"` for self-contained content
- Use `shadow="none"` on gray/subtle backgrounds

**Don't**
- Don't nest Cards inside Cards
- Don't put links or buttons directly on the card root — put them in `CardFooter`
- Don't use `shadow="lg"` for non-critical content on light backgrounds
- Don't use Card as a generic wrapper for every UI region
