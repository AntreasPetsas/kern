Layout container that groups related content into a visually distinct surface. Provides structure, not behaviour.

Use `CardHeader`, `CardBody`, and `CardFooter` for consistent internal structure.

## Usage rules

### Do
- Use Card to group a **coherent set of related information** ŌĆö a product, a person, a task.
- Use `CardHeader` / `CardBody` / `CardFooter` to establish clear information hierarchy.
- Choose the semantic element carefully: `as="article"` for self-contained content, `as="section"` for labelled groupings, `as="li"` inside a list of cards.
- Use `shadow="none"` with an explicit border when cards appear on non-white backgrounds.
- Keep Cards focused ŌĆö if a card covers two unrelated topics, split into two cards.

### Don't
- Don't **nest Cards inside Cards** ŌĆö creates visual ambiguity about hierarchy.
- Don't put interactive links or buttons directly on the card root ŌĆö put them in `CardFooter` or inline in `CardBody`.
- Don't use Card as a generic wrapper for every UI region.
- Don't use `shadow="lg"` for non-critical content on light backgrounds.
- Don't use `padding="none"` without providing your own spacing.

## Accessibility

- **Role:** depends on `as` prop (see usage rules)
- If `as="article"`, provide `aria-label` or `aria-labelledby` pointing to the `CardHeader` heading
- Ensure `CardHeader` contains a heading element (`h2`ŌĆō`h6`) for screen reader navigation
- If the card is clickable as a whole, wrap content in a single `<a>` ŌĆö don't add `onClick` to the card div
