# Design System as Living Docs вЂ” Architecture Plan

## 1. Goal

A design system (tokens, components, usage rules) that serves **three consumers equally**:

1. **LLM via context paste** вЂ” a single flat file dropped into a prompt.
2. **LLM via Skill** вЂ” file-based, progressive disclosure (`SKILL.md` + references).
3. **Human devs** вЂ” clone/install, plug-and-play, living docs site.

## 2. Core Principle

**One machine-readable source of truth, multiple generated views.**

Tokens and component specs are authored **once**. A build step emits every downstream artifact (CSS vars, Tailwind preset, TS types, docs data, Skill files, LLM context bundle). Nothing is authored twice. This is what keeps three consumers in sync instead of rotting apart.

## 3. Decisions (Locked)

| Decision | Choice | Reason |
| --- | --- | --- |
| v1 scope | Tokens + rules + 3вЂ“5 reference components | Components are *examples the LLM learns from*, not a production library. Best effort-to-value ratio. |
| Core | Tokens-only, framework-agnostic JSON | The real product; serves all consumers equally. |
| Reference components | React + plain CSS custom properties | Most legible to devs and LLMs; no framework noise; highest model-generation reliability. |
| Styling | CSS custom properties (not Tailwind/CSS-in-JS inside components) | Readable "token в†’ style" mapping. Tailwind preset still *shipped* as a generated artifact. |
| Docs site | Astro + Starlight, imports real components | Purpose-built for content/docs; MDX native; islands for interactive previews; Storybook skipped in v1. |
| Token format | W3C DTCG (`$value` / `$type`) | Tooling compatibility; increasingly what LLMs recognize. |
| Build tool | Style Dictionary (or small custom script) | Standard token transformation pipeline. |
| npm publishing | Public npm under `@org/design-system` scope | Public registry; no private infra needed. |
| Dark mode / theming | Deferred to v2 | Out of v1 scope; token layer will be structured to support it later without breaking changes. |
| Versioning + changelog | Conventional Commits + Changesets | Automated per-package versioning; changelog generated from commit history; all generated artifacts versioned together in lockstep. |

**Not in v1:** full component library (30+), Storybook, multi-framework outputs, dark mode/theming.

## 5. Plug-and-Play Paths (all layered, one source)

| Path | Consumer | Install |
| --- | --- | --- |
| `npm install` token pkg + Tailwind/CSS artifacts | Production devs | `npm i @org/design-system` |
| `git clone` + copy `skill/` | LLM Skill | File-based, clone is natural install |
| Single CSS-vars file | Zero-build consumers | Copy `build/css/variables.css` |
| `context-bundle.md` | LLM prompt paste | Copy/paste |

All four are generated from the same source; "all of the above" costs build config, not duplicated authoring.

## 6. Source-of-Truth Schemas (to draft next)

### 6a. Tokens (W3C DTCG)

- `$value`, `$type`, `$description` per token.
- Semantic layer over primitive layer (e.g. `color.primitive.blue.500` в†’ `color.semantic.action.primary`).
- Token layer structured to support dark mode/theming in v2 without breaking changes (semantic aliases are the extension point).

### 6b. Component spec (one file в†’ React + docs + Skill)

A `*.spec.json` must contain enough to generate all three views:

- `name`, `description`, `category`
- `props` (name, type, default, description)
- `variants` / `states`
- `tokensUsed` (which design tokens it consumes)
- `usageRules` (do / don't)
- `accessibility` (roles, keyboard, ARIA)
- `examples` (code snippets + rendered previews)

- Skill files use **progressive disclosure** (large system, low context cost).
- Context bundle is the **opposite**: one dense flat dump for paste-in.
- Both regenerate on token/spec change. No manual sync.

## 8. Build Order (v1)

1. Define `tokens/` schema (DTCG) вЂ” primitive + semantic layers.
2. Wire Style Dictionary в†’ emit `build/css`, `build/tailwind`, `build/ts`.
3. Define component `spec` schema; author 3вЂ“5 specs.
4. Build React reference components from specs (CSS vars).
5. Generator: specs + tokens в†’ `skill/` and `llm-context/`.
6. Living docs site importing real components + rules.
7. Author usage rules (global + per-component do/don't).
8. Package for npm; document the four install paths.
